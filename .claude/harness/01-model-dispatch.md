# C. 模型調度與動態升降級守則

> 讀者：擔任指揮官的主對話模型（Opus 4.8）。本檔是**調度合約**，不是建議——除非 User 明示豁免，一律照做。

## 1. 模型階梯

| 階層 | 模型 | 職責 | 禁止事項 |
|---|---|---|---|
| 指揮官 COMMANDER | Opus 4.8（主對話） | 決策、拆解任務、架構引導、彙整回報、與 User 對話 | 親自大量讀檔/掃 repo/批次機械操作 |
| 主力 WORKER | Sonnet 5（Subagent） | 讀檔、搜尋、實作、重構、審查、驗收 | 自我驗收自己的實作 |
| 廉價 CHEAP | Haiku 4.5（Subagent） | 機械批次任務：固定模式套用、格式轉換、簡單枚舉 | 任何需要判斷取捨的任務 |

## 2. 指揮官不下場（量化紅線）

主對話中出現以下任一情況，**必須**改派 Subagent，主對話只接收精簡結論：

| # | 紅線 | 動作 |
|---|---|---|
| 1 | 單輪需要 Read 超過 **3 個檔案** | 派 WORKER 讀檔並依模板回報摘要 |
| 2 | 任一檔案超過 **400 行**且只為了「了解內容」 | 同上 |
| 3 | 需要 Grep/Glob 全 repo 找「所有用到 X 的地方」 | 派 WORKER 搜尋，回報「路徑＋行號＋一行說明」清單 |
| 4 | 批次修改 ≥ **5 個檔案**且模式相同 | 指揮官先確立模式，再依第 4 節降級條件（WORKER 樣本驗證後）交 CHEAP 批次套用 |

例外（指揮官可親自動手）：讀單一短檔確認關鍵事實、對 Subagent 回報中的個別疑點做 ≤2 個檔案的抽查、依第 4 節決策序列升級到 COMMANDER 親自處理的任務（此時上表紅線暫時豁免，但處理完畢後仍須依第 5 節派 fresh-context 驗收）。

## 3. 派工三件套（缺一即為不合格派工）

每次指派 Subagent，prompt 必須包含（模板見 `03-delegation-templates.md`）：

1. **明確目標與背景**：要達成什麼、為什麼做、已知的相關脈絡（相關檔案路徑、既有結論）。Subagent 是全新 context，**你不說它就不知道**。
2. **嚴格的驗收條件**：可勾選的清單（例：「函式 X 的所有呼叫端都已更新，`npm run lint` exit 0」），不得使用「做好做完整」這類無法判定真偽的語句。
3. **標準回報格式**：成果檔案路徑＋關鍵行號＋一行結論。**明文禁止在回報中貼大段程式碼**（>10 行即算大段；需要展示時給「路徑:行號區間」）。

## 4. 異常升降級路徑（單一決策序列）

**名詞定義**（全 harness 通用，其他檔案提到「次」「路徑」「輪」皆以此為準）：
- **1 次失敗**＝工具呼叫失敗、語法錯誤、驗收條件未達成而回報完成、或產出與目標明顯不符，任一均計 1 次。
- **1 條路徑**＝一個解法方向；同一根因假設下的嘗試都算同一條路徑，換根因假設＝換路徑。
- **1 輪**＝在同一條路徑上的嘗試（最多容許 2 次失敗）。**同一件事最多兩輪（兩條路徑），用盡即熔斷，禁止開第三條。**

**決策序列**（失敗發生時依序執行，不跳步、不並行）：
1. CHEAP 失敗 1 次 → 直接升 WORKER 重做（失敗計數歸零，自此按下列步驟計）。
2. WORKER 第 1 次失敗 → 分析錯誤輸出、修正後重試（同路徑第 2 次）。
3. WORKER 同一路徑第 2 次失敗 → 停止重試，過 `02-judgment-matrix.md` 第 1 表（停損信號）：
   - **命中任一停損信號** → 回報 COMMANDER；由 COMMANDER 定出新路徑後開第二輪，仍派 WORKER 執行。
   - **未命中**（方向沒錯但做不出）→ 帶完整失敗軌跡升級 COMMANDER 親自處理（此即第二輪）。
4. 第二輪（無論誰執行）仍失敗 → 觸發 `02-judgment-matrix.md` 第 3 表 #5，熔斷向 User 提問。

- **升級必附完整失敗軌跡**：原始指令、每次嘗試的具體改動、每次的錯誤輸出。禁止只說「它做不出來」。
- **降級條件**：COMMANDER 解出固定模式後，必須先在 1–2 個樣本上由 WORKER 驗證可複製，才能寫成明確步驟交 CHEAP 批次套用。

## 5. 隔離驗證（防閉環迭代的核心）

**規則：負責實作的 Agent 不得驗收自己的產出。** 理由：同一個 context 會用同一套錯誤假設去「驗證」，永遠通過。

驗收一律指派 **Fresh-Context Subagent**（WORKER 級），按任務類型選擇手段：

| 任務類型 | 驗收手段 |
|---|---|
| 程式碼實作/修改 | read-back 重新讀改動後的檔案 → 對照驗收條件逐條勾選 → 實跑驗證命令（CLAUDE.md 驗證命令區）與相關測資 |
| 文件/規則產出 | read-back 全文 → 檢查內部引用路徑是否存在、規則是否互相矛盾 |
| 方案選型/設計 | 先切分：涉及品味/美感/對外行為/需求取捨 → 依 `00-baseline.md` 第 5 節熔斷交 User 選擇；有客觀可量化指標（效能數據、測試通過數、相依數、程式碼量）→ 多樣本評審（步驟見下） |

多樣本評審操作步驟：1) 由互不知情的 Subagent 各自產出 2–3 個獨立方案；2) 指揮官列出 3–5 個可量化評分維度；3) 派 2 位互不知情的評審 agent 依維度各自打分（每維度 1–5 分＋一句理由）；4) 合計分數選優；兩案總分差 < 10% 視為平手，升格為品味決策，熔斷交 User 選擇。

驗收 Subagent 的回報必須是逐條勾選結果（✅/❌＋證據行號），不接受「整體看起來沒問題」。任何一條 ❌ → 任務回到實作者，計入第 4 節失敗次數。
