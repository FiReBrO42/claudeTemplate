## 一、三件使用者沒問、但我認為最關鍵的事

### 1. 這套制度的生死線在「前三週的例外」
制度不是被推翻的，是被例外磨死的。第一次「這個小改動就不派驗收了吧」、第一次「驗證命令還沒建，先標完成吧」，都不會立刻出事——正因為不會立刻出事，例外就會變成慣例。**技術棧確定當天，第一件事是填 CLAUDE.md 的驗證命令區**（`00-baseline.md` 第 4 節有完整補完清單）；在那之前，請把每一個「已實作、未驗證」的回報當真，不要腦內折算成「其實就是完成了」。

### 2. 指揮官的 Context 紀律是整套 Harness 的承重牆
其他規則壞了會出 bug，這條壞了會**讓所有規則一起失效**：主對話一旦被大量檔案內容灌爆，指揮官會忘記自己訂過的規則、忘記任務目標、開始重複已做過的事——而且劣化是無感的，你不會收到任何報錯。所以 `01-model-dispatch.md` 第 2 節的紅線是量化的（3 檔/400 行）：不是建議值，是讓弱模型「數得出來」的物理界線。覺得紅線太緊時，正確動作是提案修改數字，不是默默越線。

### 3. 知識系統的預設命運是爛掉，方向有二
- **餓死**：沒人寫。對策已內建——踩坑當場落檔是 CLAUDE.md 核心規則第 3 條，派工模板也要求回報落檔情況。
- **撐死**：什麼都寫、從不精簡，lessons 變垃圾場，弱模型每次開工讀 800 行噪音。對策是 `04-knowledge-protocol.md` 第 6 節的精簡觸發，但**觸發後的裁決需要 User**——請 User 把「收到精簡提案就花 10 分鐘處理」當成固定義務，這是整套制度中少數無法自動化的環節。

## 二、弱模型長期運作下，這套制度最可能的退化方式與預防

| 退化方式 | 早期徵兆 | 預防/矯正 |
|---|---|---|
| 隔離驗證被跳過（實作者自報完成，指揮官照單全收） | 回報中沒有驗收 Subagent 的逐條勾選 | 指揮官收報時只認格式：無勾選清單=未驗收，退回 |
| 派工模板退化成走過場 | 模板欄位填著「見上文」「同前」「做好即可」 | 模板鐵律：「[提示文字]」佔位符填不實不發出；驗收條件必須可判定真偽 |
| 規則檔被「順手」改壞 | `.claude/harness/` 出現未經 User 同意的結構性 diff | 權限分級（`04-knowledge-protocol.md` 第 5 節）＋ User 在 review 時多看一眼 harness 目錄的 diff |
| lessons 條目劣化成「要小心 X」式廢話 | 新條目缺〔根因〕〔重現〕欄位 | 格式即門檻：缺欄位=未落檔，不算完成 |
| general.md 與 harness-global 分歧 | 兩邊條目數對不上 | 記住單一方向：repo 為權威，回填本機；每次雙寫當場完成，不積欠 |
| CLAUDE.md 重新膨脹 | 有人往「最小核心規則」區加第 4、5、6 條 | 路由中心原則：新規則一律獨立成檔＋路由表加一行 |

## 三、harness-global 備份提醒（重要）

`~/.claude/harness-global/` 是你跨專案的個人通用教訓累積器，**不進任何 git repo**——換電腦、重灌系統時它不會自動跟著走，多年累積可能一夕蒸發。建議二選一，現在就做：
1. `cd ~/.claude/harness-global && git init`，推到一個**私有** GitHub repo（推薦，順便獲得歷史紀錄）；
2. 把它加入你既有的定期備份（Time Machine 已涵蓋家目錄的話，確認它在備份範圍內）。
