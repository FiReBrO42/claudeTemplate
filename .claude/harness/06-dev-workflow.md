# 開發流程閉環（Dev Workflow）

> 定位：制度化閉環（00–05）回答「誰來做、怎麼派、怎麼判斷、怎麼沉澱」；本檔回答「**一件開發任務從需求到交付回顧，要經過哪些站、每站的進出條件是什麼**」。
> 技術棧無關：本檔不寫死任何棧細節。所有棧相關具體值（命令、結構、規範）一律查 `07-stack-profile.md`；驗證命令的單一事實來源是專案根 CLAUDE.md「驗證命令」區。

## 1. 閉環全景

```
需求進入 → 站0 規模分級 ─┬─ S ──────────────────→ 站3 實作 → 站4 輕量驗證 → 站5 交付 ──┐
                          ├─ M → 站1 需求釐清 ────→ 站3 實作 ⇄ 站4 驗證 → 站5 審查交付 → 站6 回顧 ─┤
                          └─ L → 站1 → 站2 設計 ──→ 站3 實作 ⇄ 站4 驗證 → 站5 審查交付 → 站6 回顧 ─┤
                                                                                                    │
        lessons（general.md / project-lessons.md）←──────── 教訓落檔 ←──────────────────────────────┘
                          │
                          └──→ 下一件任務的「開工前必讀」把教訓讀回 → 環閉合
```

閉環的意義：站 6 產出的教訓在下一件任務開工時被讀回，同一個坑不踩第二次。**任何一站被靜默跳過，環就斷**——跳站必須在回報中明說並給依據（如「S 級依規免站 2」）。

## 2. 站 0：規模分級（COMMANDER 派工前必做）

以「預估改動檔案數」＋「是否新增/變更程式行為」二維判定，**符合任一較高級的條件就向上取級**：

| 級別 | 判準 | 必經站 |
|---|---|---|
| S 微型 | 預估 ≤2 檔案 **且** 不新增/變更程式行為（文案、註解、樣式參數、設定值） | 3 → 4（輕量）→ 5 |
| M 標準 | 預估 3–10 檔案，或任何新增/變更程式行為 | 1 → 3 → 4 → 5 → 6 |
| L 大型 | 預估 >10 檔案，或符合任一：改變被 ≥2 處引用的公共函式/元件/API 簽名；變更持久化資料格式或跨模組共用型別；驗收條件中 ≥3 條涉及使用者可見行為變更 | 1 → 2 → 3 → 4 → 5 → 6 |

- 分級結論寫進派工單【分級】欄（03 模板 2/3 已內建），一行即可：「本任務分級：M。依據：預估 5 檔＋新增行為」。
- S 級不設獨立站 1：填寫派工模板（03）本身就是最小需求釐清。
- **實作中升級**（本檔獨立規則；與 `02-judgment-matrix.md` 第 1 表 #5 的「3 倍基準」停損信號各自獨立、皆須遵守）：實際改動檔案數超過原分級上限 → 停手回報 COMMANDER，重新分級並補走漏掉的站。
- 拿不準 S 還是 M → 一律按 M 走；拿不準 M 還是 L → 一律按 L 走。往低分級是省流程的優化，不是預設權利。

## 3. 各站定義

每站四件事：進站條件／做什麼／出站條件（**全勾才能離站**）／產物。

### 站 1 需求釐清（COMMANDER）
- 進站：收到 User 需求或既有待辦。
- 做什麼：把需求翻譯成「可勾選的驗收條件清單」；過 `02-judgment-matrix.md` 第 3 表 #1 二義性檢查，命中 → 先熔斷問清楚再開工。
- 出站條件：
  - [ ] 每條驗收條件都可判定真偽（有具體輸入輸出，無「做好做完整」式語句）
  - [ ] 已讀 lessons 兩檔，相關舊坑已寫進派工單【背景】欄
- 產物：任務卡（＝03 模板的已填實版）。

### 站 2 設計（僅 L 級）
- 進站：L 級、站 1 出站條件全勾。
- 做什麼：產出 2–3 個候選方案，依 `01-model-dispatch.md` 第 5 節切分——可量化比較的 → 多樣本評審；涉及品味/取捨的 → 熔斷交 User 選。
- 出站條件：
  - [ ] 選定方案已落檔，內含：選定理由、每個落選方案一行否決理由、預估影響檔案清單
  - [ ] 預估影響檔案數與站 0 分級一致（不一致 → 回站 0 重分級）
- 產物：`docs/plans/YYYY-MM-DD-<任務slug>.md`（進 git；目錄首次使用時建立）。**後續派工單【背景】必須引用此檔路徑**——設計沒被實作引用，等於沒設計。

### 站 3 實作（WORKER，依 03 模板 2 或 3 派工）
- 進站（按分級）：S 級＝03 模板佔位符全數填實；M 級＝站 1 出站條件全勾；L 級＝站 1＋站 2 出站條件全勾。
- 做什麼：紅綠循環——先寫「該紅時會紅」的測試 → 實作至轉綠（測試框架未建 → 依 03 模板 2 的 fallback，完成上限為「已實作、未驗證」）。
- 硬規則：**WORKER 需要新增 runtime 依賴（新套件）時 → 停止實作，回報 COMMANDER（附一行理由與一行替代方案），由 COMMANDER 依 `02-judgment-matrix.md` 第 3 表向 User 熔斷提問**。新增 dev 依賴可先行，但必須在回報中標明。此禁令已內建於 03 模板 2/3 的【邊界】欄，隨派工單傳達給 WORKER。
- 出站條件：03 模板 2/3 的驗收條件全勾（含 `02-judgment-matrix.md` 第 2 表自查）。
- 產物：程式碼改動＋依模板格式的回報。

### 站 4 驗證（fresh-context WORKER，規則見 `01-model-dispatch.md` 第 5 節）
- 進站：站 3 出站條件全勾。
- S 級輕量版（三項勾選即可）：
  - [ ] read-back 全部改動檔案
  - [ ] 實跑驗證命令 exit 0（驗證命令區為空 → 標記「已實作、未驗證」）
  - [ ] 無夾帶與任務無關的改動
- M/L 完整版：01 §5 對應手段＋`02-judgment-matrix.md` 第 2 表逐條。
- **驗證命令區為空時的通則（S/M/L 皆適用）**：驗證命令該項標記「已實作、未驗證」，視同通過該單項（不計 ❌），但任務**整體狀態**降為「已實作、未驗證」帶入站 5。
- 出站條件：驗收回報全 ✅（含依上一條視同通過者）。任何 ❌ → 退回站 3，計入 01 §4 失敗次數。
- 產物：驗收回報（逐條勾選＋證據行號）。

### 站 5 審查與交付（COMMANDER 主持；審查另派 fresh-context WORKER，commit 由 COMMANDER 執行）
- 進站：站 4 出站條件全勾。
- M/L：先過 03 模板 4 審查（fresh-context，不得由站 3 或站 4 的執行者兼任）→ 通過才交付。
- S：站 4 通過即交付（審查已併入輕量驗證）。
- 交付動作：commit——規範查 `07-stack-profile.md` 欄位 7；未填 → 預設：首行 ≤50 字摘要＋空行＋動機一段。是否推送/提測/PR 查 07 欄位 8；未填 → **只 commit 不推送**，在回報中提醒 User。
- 任務整體狀態為「已實作、未驗證」時仍可 commit，但 commit 訊息與交付回報都必須以「已實作、未驗證」開頭，並把「建立驗證命令」列為最高優先待辦回報 User。
- 出站條件：
  - [ ] 已 commit 且訊息符合規範
  - [ ] 交付回報已送出（改了什麼、驗證證據或「已實作、未驗證」聲明、殘留風險各一段）
- 產物：commit＋交付回報。

### 站 6 回顧（COMMANDER，規則見 `04-knowledge-protocol.md` 第 7 節）
- M/L 必做；S 僅在踩坑時做。
- 出站條件：
  - [ ] 「有沒有還沒落檔的坑？」已自問自答（有 → 已依 04 分流落檔）
  - [ ] 「有沒有哪條規則造成摩擦？」已自問自答（有 → 已依 04 §5 提案）
- 產物流回 lessons——**閉環在此閉合**。

## 4. 本流程的三大反模式與防治

1. **流程通膨**：對 S 級任務跑六站全流程，Token 燒在儀式感上。防治：站 0 分級先行，S 級三站封頂。
2. **跳站交付**：「程式碼都寫好了，直接 commit 吧」——跳過站 4/5 省下的時間，遲早以 10 倍返工成本償還。防治：站 5 出站條件要求附驗證證據；沒有證據＝沒過站，指揮官收報時只認格式。
3. **設計文檔孤兒化**：L 級寫了 plans 檔，實作派工單卻不引用它，寫了等於沒寫。防治：站 2 產物規則強制派工單引用路徑；站 5 審查時必核對「實作與設計的偏離點」。
