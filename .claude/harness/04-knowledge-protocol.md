# F. 知識迭代與反思協議

> 用途：定義弱模型如何安全地累積教訓、以及如何在**不弄壞制度**的前提下更新這套 Harness。
> 核心機制三個詞：**分流**（專屬 vs 通用）、**雙寫**（通用教訓寫兩份）、**播種**（新專案繼承通用教訓）。

## 1. 分流判準（寫入任何踩坑/教訓前必做）

**一句話準則：內容提到本專案的檔名／函式／API／業務規則／目錄結構 → 專屬；換任何專案仍成立 → 通用。**

| 判定 | 寫入位置 | 正例 | 反例（常見誤判） |
|---|---|---|---|
| 通用 | **雙寫**：`lessons/general.md` ＋ `~/.claude/harness-global/general.md` | 「某 CLI 工具在路徑含空白時會靜默失敗，需加引號」——換任何專案都成立 | 「本專案 `UserStore.fetchProfile` 回傳缺 avatar 欄位」——綁定本專案程式碼，**不算通用** |
| 專屬 | `lessons/project-lessons.md` | 「本專案 `UserStore.fetchProfile` 回傳缺 avatar 欄位，呼叫端需自行補預設值」 | 把「async 函式忘記 await 會吞錯誤」寫進專屬檔——這是通用知識，寫專屬檔就播種不出去了 |
| 無法判定 | 預設寫入 `project-lessons.md`，條目末尾標註 `〔待確認是否可上升為通用〕` | — | 因為猶豫就不寫——**不寫是最大的錯**，寫錯位置是小錯 |

## 2. 雙寫規則（通用教訓專用）

1. 寫入 `lessons/general.md`（進 git，**團隊權威版**，組員立即可見）。
2. 同時鏡射一條到 `~/.claude/harness-global/general.md`（本機個人累積器，供未來新專案播種），並在條目加註〔來源專案：invest〕；鏡射後在該目錄執行 `git add -A && git commit`（harness-global 自 2026-07-04 起為獨立 git repo）。
3. 兩份不一致時：**以 repo 的 `lessons/general.md` 為準，回填本機**。方向永遠是 repo → 本機，不得反向覆蓋 repo。
4. 若在無法存取家目錄的環境（如 CI）中產生通用教訓：先寫 general.md，並在條目加註 `〔待鏡射至 harness-global〕`，下次本機 session 補鏡射後移除該註記。
5. 發現**本機有、repo 無**的條目：不得刪除本機條目，視為「待審提案」——在對話中列給 User，經同意後補進 repo 的 general.md（這不算「上升為通用」審批，因為條目本來就是通用，只是漏了雙寫）。

## 3. 播種規則（開新專案/套範本時）

1. 讀取 `~/.claude/harness-global/general.md` 全部條目。
2. 與新專案既有的 `lessons/general.md`（若有）合併去重（同一根因只留一條，取描述較完整者）。
3. 寫入新專案 `lessons/general.md`，檔頭註明播種日期與來源。
4. **由既有專案複製範本開新專案時**：`project-lessons.md` 必須清空——專屬踩坑不得跨專案沿用（別的專案的檔名/API 在新專案是純噪音，還會誤導弱模型）。

## 4. 踩坑紀錄統一格式

每條教訓一律用以下欄位（缺欄位的條目視為不合格，補齊前不得算「已落檔」）：

```markdown
### [一句話標題]
- 〔症狀〕觀察到的錯誤現象
- 〔觸發條件/重現步驟〕什麼情況下會發生
- 〔根因〕真正的原因（不是表象）
- 〔正確做法〕下次遇到該怎麼做（可執行的指令/步驟，不是「要小心」）
- 〔判準〕專屬 ／ 通用 ／ 待確認是否可上升為通用
- 〔日期〕YYYY-MM-DD
```

**反例**：「i18n 的 key 要記得同步，不然會出錯。」——無症狀、無重現、無根因、「記得」不可執行。這種條目寫了等於沒寫。

## 5. 權限分級

**「結構性修改」的操作型定義**（符合任一即是，需 User 同意）：新增/刪除章節或表格、變更表格欄位結構、變更任何量化門檻數字（紅線、失敗次數上限、精簡門檻）、改寫既有規則的語意。

**預授權動作白名單**（不算結構性修改，弱模型可自行執行）：
- 填入規則檔中預留的 TODO/佔位欄位（如 CLAUDE.md 驗證命令區、`00-baseline.md` 技術棧欄位）
- 勾銷 `00-baseline.md` 第 4 節補完清單項目
- 在 CLAUDE.md 路由表／MCP 登記區**新增一行**（不改既有行）
- 修正錯字與失效路徑

**備份規則**：對制度規則檔（CLAUDE.md 與 `.claude/harness/` 下全部規則檔，即 README 與 00–07）執行任何修改（含預授權動作）前，先 `cp 檔案 檔案.bak`（`*.bak` 已列入 .gitignore）；該輪工作結束、修改確認無誤後刪除 .bak。

| 操作 | 弱模型可自行執行？ |
|---|---|
| 在 lessons 兩檔**新增**符合格式的條目 | ✅ 可，當場落檔 |
| 修正 lessons 既有條目的錯字/補充重現步驟 | ✅ 可 |
| 把某條專屬教訓**上升為通用**（搬進 general.md＋鏡射本機） | ❌ 需先徵得 User 同意 |
| **刪除** lessons 條目 | ❌ 需 User 同意（可先標註〔疑似過時〕） |
| 制度規則檔（CLAUDE.md 與 harness 下全部規則檔：README、00–07）的**結構性修改** | ❌ 需 User 同意；發現規則問題時，在對話中提案並引用原文，不動檔案 |
| 制度規則檔的錯字修正、失效路徑更新 | ✅ 可，但需在當輪回報中明列改了哪一行 |

## 6. 自動精簡觸發

任一 lessons 檔滿足以下**任一**條件時，當前 session 必須向 User 提案精簡（提案內容：合併哪些、上升哪些、預計刪哪些）：

- 條目數 > **40 條**，或
- 總行數 > **800 行**

精簡的三個動作：**合併重複**（同根因合一條）、**抽象共通模式**（多條專屬踩坑指向同一深層原因 → 提煉為一條通用教訓，上升需 User 同意）、**標記過時**（技術棧/架構已變的條目標〔疑似過時〕，刪除需 User 同意）。

## 7. 定期反思（輕量，不設排程強制）

每完成一個「多 Subagent 協作的大任務」後，COMMANDER 花一輪自問：
1. 這次有沒有踩坑但還沒落檔的？→ 有就當場落檔。
2. 有沒有哪條 harness 規則在這次執行中造成明顯摩擦？→ 有就依第 5 節權限提案，不悶著。
