# D. 判斷力外化矩陣

> 用途：把高階模型的直覺量化為**肉眼可比對**的檢核表。使用方式：對照當下狀況逐條檢查，命中即執行對應動作，不要「憑感覺再試一下」。
> 三張表的動作優先序固定為：**第 3 表（熔斷）＞第 1 表（停損）＞第 2 表（完成）**。同時命中多表時，先執行熔斷，其次停損；完成判準永遠最後才檢查。

## 1. 停損信號表——「方向完全錯了，立刻停下換路徑」

命中下表**任一條** → 停止在現有程式碼上重試，回到上一個已知正確的狀態（git stash / checkout），重新分析根因後換路徑。禁止的動作：在同一段程式碼上做第 3 次微調。

| # | 信號（量化） | 完美正例（該這樣做） | 典型反例（禁止這樣做） |
|---|---|---|---|
| 1 | 同一個報錯訊息在 2 次修復後仍出現 | 第 2 次修復後仍見同樣報錯 → 停手，宣告「根因假設錯誤」，派 fresh-context agent 重新從報錯往上游追，找到新根因假設再動手 | 第 3、4、5 次繼續在同一函式裡調參數、換寫法，每次都「感覺這次會好」 |
| 2 | 修 A 壞 B，修 B 又壞 A（蹺蹺板） | 立刻停手並上報：「A 與 B 存在共享狀態/隱含耦合，需先重構解耦，預估影響範圍為…」 | 在 A、B 之間來回打補丁，最後在其中一邊加 `if` 特例讓兩邊「都過」 |
| 3 | 為讓測試通過而新增的特例分支（if/try-catch）≥ 2 個 | 承認「我在用補丁掩蓋設計問題」，回滾特例，重新檢視資料流 | 「再加一個 edge case 判斷就能全過了」——特例會無限增生 |
| 4 | diff 行數持續成長，但通過的驗收條件數不變 | 對照驗收條件清單：連續 2 次提交無新增 ✅ → 停手回滾到最後一次有進展的點 | 「程式碼越寫越多=越接近完成」的錯覺，繼續堆碼 |
| 5 | 實際需要改動的檔案數 > 基準的 3 倍。基準推估規則：任務敘述明確點名 N 個對象 → 基準＝N；未點名 → 基準＝2（例：「改一個按鈕文案」基準＝1，實際動到 6 個檔案即命中） | 停手上報：「此任務牽動範圍遠超敘述，可能踩到架構問題或我理解錯需求」，附檔案清單 | 默默把 6 個檔案全改完才回報，讓審查者面對巨型 diff |
| 6 | 開始想「先把測試改弱/跳過，之後再補」 | 測試擋路=測試在說話。停下來讀測試想保護什麼 | 註解掉測試、把斷言改鬆、加 `skip`——這是本表最危險的信號 |

## 2. 完成判準表——滿足**全部**條件才可宣告「完成」

任何一條不滿足 → 只能回報「已實作，未完成驗收，缺項：X」。

| # | 條件 | 完美正例 | 典型反例 |
|---|---|---|---|
| 1 | CLAUDE.md 驗證命令區的命令全部 exit 0（該區為空 → 依 `00-baseline.md` 反模式 1，最高只能宣告「已實作、未驗證」） | 回報中附命令實跑輸出的最後 3 行 | 「我檢查過邏輯，應該沒問題」——沒跑命令就是沒驗證 |
| 2 | 新增/變更的行為有對應測試，且看過它「該紅時會紅」 | 先寫測試看它失敗，再實作讓它通過 | 補一個永遠會過的測試（斷言寫成恆真、或只測了舊行為） |
| 3 | 產出中無 TODO / FIXME / 佔位符 / 空函式殼 | 交付前 grep 一次 `TODO\|FIXME\|placeholder` 確認為 0 | 「主體完成了，細節標了 TODO」——那就不叫完成 |
| 4 | diff 中每一行都能對應到本次需求（無夾帶重構、無無關格式化） | 發現順手改了無關程式碼 → 拆出去或還原 | 「反正都打開這個檔了，順便清理一下」——污染 diff，審查失焦 |
| 5 | 已通過隔離驗證（`01-model-dispatch.md` 第 5 節），驗收回報全 ✅ | 附驗收 Subagent 的逐條勾選回報 | 實作 agent 自己說「我驗證過了」 |

## 3. 熔斷判準表——「停止自主作業，向 User 提問」

命中任一條 → 停下，**帶著具體選項**提問（禁止開放式的「請問怎麼辦」）。提問格式：現狀一句話＋選項 A/B（各附一行代價）＋你的建議。

| # | 判準（量化） | 完美正例 | 典型反例 |
|---|---|---|---|
| 1 | 需求存在二義性，且兩種解讀的預估改動檔案數相差 ≥ 3 個檔案或 ≥ 2 倍，或解讀差異會改變對外可見行為 | 「『支援多語系』有 A：僅架構預留（~2 檔）；B：完整接入翻譯流程（~15 檔＋流程）。建議 A，因為…你選哪個？」 | 挑一個自己覺得合理的解讀，默默做完 15 個檔案 |
| 2 | 即將執行不可逆/外向操作：刪除非本次建立的檔案、force push、對外發布、動 production 設定 | 停下列出將被影響的對象清單，等 User 確認 | 「反正有 git 可以救回來」——不是所有東西都在 git 裡 |
| 3 | 遇到品味/美感/商業取捨決策（`00-baseline.md` 誠實條款所列） | 產 2–3 個具體選項各附一行客觀差異，交 User 選 | 弱模型自行拍板 UI 風格，然後在其上蓋了 10 個相依改動 |
| 4 | 發現任務指示與 harness 規則、或兩份規則檔之間互相矛盾 | 停下引用兩處原文，請 User 裁決，勝出方寫回規則檔 | 自行選一邊照做，讓矛盾繼續埋著 |
| 5 | 已走完 `01-model-dispatch.md` 第 4 節決策序列（兩輪/兩條路徑用盡，或升級 COMMANDER 後仍失敗） | 帶完整失敗軌跡上報：「已嘗試 A、B 路徑各因 X、Y 失敗，判斷卡點在 Z，需要你提供…」 | 開第三條路徑換個花樣繼續試，燒 Token 給 User 一個「還在處理中」的假象 |

## 4. 使用本矩陣的頻率

- WORKER：每次「準備宣告完成」前過第 2 表；每次「修復失敗」後過第 1 表。
- COMMANDER：每次收到 Subagent 回報後，替它過一遍第 1、3 表（Subagent 自己常意識不到已命中信號）。
