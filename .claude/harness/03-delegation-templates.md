# E. 標準化派工 Prompt 模板

> 用法：指揮官（Opus 4.8）複製對應模板 → 把 `[ ]` 填實 → 作為 Subagent 的完整 prompt 發出。
> 鐵律：模板中「[中文提示文字]」式**佔位符**必須全部填實才能發出——填不出來代表你自己還沒想清楚，先想清楚再派工。「- [ ]」**核取方塊**則是給對方逐條勾選的驗收清單，發出時保持未勾，兩種符號不可混淆。
> 所有模板已內建「派工三件套」（目標背景／驗收條件／回報格式，見 `01-model-dispatch.md` 第 3 節）。

---

## 模板 1：搜尋研究（WORKER 或 CHEAP）

```
【任務】在本 repo 中調查：[要回答的具體問題，一句話]
【背景】[為什麼要查；已知線索，如：相關檔案可能在 src/xxx/、關鍵字為 yyy]
【範圍】搜尋範圍限定：[目錄/檔案類型]；明確排除：[如 node_modules、dist、測試快照]
【驗收條件】
- [ ] 找到全部匹配項，或明確回報「掃描了哪些範圍、確認不存在」
- [ ] 每個發現都附「檔案路徑:行號」
【回報格式】（總長 ≤ 30 行）
1. 一句話結論
2. 發現清單：`路徑:行號` ＋ 各一行說明
3. 未能確認的部分與原因
禁止：貼大段程式碼（>10 行）；對發現做本任務未要求的延伸建議。
```

## 模板 2：功能實作（WORKER）

```
【任務】實作：[功能的一句話描述]
【背景】[需求來源與目的；相關既有程式碼路徑；必須遵循的既有模式，如：參照 src/xxx 的寫法]
【分級】本任務分級：[S/M/L]，依據：[一句話，分級規則見 06-dev-workflow.md 站 0]
【邊界】允許改動：[目錄/檔案清單]；禁止改動：[清單，預設含測試的斷言、公共 API 簽名]；禁止新增 runtime 依賴——需要時停手回報（附一行理由與一行替代方案），由 COMMANDER 徵求 User
【開工前必讀】.claude/harness/lessons/ 下兩個檔案（避免重踩已知坑）
【驗收條件】
- [ ] [具體行為 1：輸入 X 時得到 Y]
- [ ] [具體行為 2]
- [ ] 驗證命令全部 exit 0：[命令；CLAUDE.md 驗證命令區為空則寫「無——完成上限為『已實作、未驗證』」]
- [ ] 新行為有對應測試，且該測試在實作前會失敗［測試框架尚未建立則填「無——完成上限為『已實作、未驗證』」，並在回報中把建立測試框架列為待辦］
- [ ] 自查 .claude/harness/02-judgment-matrix.md 第 2 表 #1–#4 通過（#5 隔離驗證由後續驗收 Subagent 完成，不在自查範圍）
【回報格式】（總長 ≤ 25 行）
1. 一句話結果（完成 / 已實作未驗證＋缺項 / 卡住＋卡點）
2. 改動檔案清單：`路徑:行號區間` ＋ 各一行改了什麼
3. 驗證命令實跑輸出的最後 3 行
4. 若過程中踩坑：已依 04-knowledge-protocol.md 落檔至 [檔名]
禁止：貼大段程式碼；宣告完成但驗收條件有任一未勾。
```

## 模板 3：程式碼重構（WORKER）

```
【任務】重構：[對象] —— 目的：[可驗證的目的，如：消除 X 的重複實作／把 Y 從 Z 中解耦]
【背景】[為什麼現在重構；哪些既有行為絕對不能變]
【行為不變契約】重構前先確認測試現況：[測試命令]。重構後同一命令結果必須完全一致（不新增失敗、不減少案例數）。
【分級】本任務分級：[S/M/L]，依據：[一句話，分級規則見 06-dev-workflow.md 站 0]
【邊界】允許改動：[清單]；禁止：改動任何測試斷言、改變公共 API 簽名、順手加新功能、新增 runtime 依賴（需要時停手回報，由 COMMANDER 徵求 User）
【開工前必讀】.claude/harness/lessons/ 下兩個檔案（避免重踩已知坑）
【驗收條件】
- [ ] 重構前後測試結果一致（附前後輸出對照）
- [ ] [重構目的的量化驗證，如：X 的重複實作從 3 處變 1 處]
- [ ] diff 中每一行都服務於重構目的（02-judgment-matrix.md 第 2 表第 4 條）
【回報格式】（總長 ≤ 25 行）
1. 一句話結果
2. 前/後結構對比（各 ≤ 5 行的文字描述，非程式碼）
3. 改動檔案清單：`路徑:行號區間`
4. 測試前後輸出對照（各最後 3 行）
```

## 模板 4：程式碼審查（WORKER，必須 fresh-context，不得由實作者兼任）

```
【任務】審查以下改動：[diff 範圍，如：git diff main..feature-x／某次派工的產出]
【背景】此改動的目標與驗收條件：[貼上原派工單的目標與驗收條件——沒有這個，審查就沒有依據]
【開工前必讀】.claude/harness/lessons/ 下兩個檔案（作為「有無違反教訓」勾核依據）
【審查清單】（逐條給 ✅/❌，❌ 必附「路徑:行號」與一句話理由）
- [ ] read-back：重新讀過全部改動檔案（不是只看 diff）
- [ ] 原驗收條件逐條核對
- [ ] 實跑驗證命令：[命令]，exit 0
- [ ] 對照 .claude/harness/02-judgment-matrix.md 第 1 表：是否存在停損信號（特例分支增生、測試被改弱）
- [ ] 有無夾帶與任務無關的改動
- [ ] 有無違反 lessons 中已記錄的踩坑教訓
【回報格式】（總長 ≤ 30 行）
1. 結論：通過 / 退回（退回必附最嚴重的 1–3 個問題）
2. 逐條勾選結果＋證據行號
3. 驗證命令實跑輸出的最後 3 行
禁止：「整體看起來沒問題」這類無逐條證據的結論；因小瑕疵擋下已達驗收條件的產出（小瑕疵單獨列出，標記「不擋收」）。
```

---

## 通用附註

- 派給 **CHEAP（Haiku）** 時，額外附上「已驗證可行的操作步驟」（由 COMMANDER 或 WORKER 先在 1–2 個樣本上確立），CHEAP 只做複製套用，不做判斷。
- Subagent 回報若不符「回報格式」，指揮官退回要求重報，**不要自己動手整理**——整理一次，格式規則就死了。
